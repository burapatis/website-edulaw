# คู่มือตรวจสอบแหล่งข้อมูลทางการด้วยมือ (Source Verification Worksheet)

คู่มือนี้ใช้คู่กับ:

- `docs/SOURCE_VERIFICATION_AUDIT.md` — ตารางสถานะแหล่งข้อมูลของทุกหน้า (สร้างอัตโนมัติ)
- `docs/PRIORITY_LAW_SOURCE_CHECKLIST.md` — เช็กลิสต์เรียงตามลำดับความสำคัญ (สร้างอัตโนมัติ)
- `SOURCE_REVIEW_LOG.md` — บันทึกผลการตรวจถาวร (กรอกด้วยมือ)
- `LEGAL_REVIEW_CHECKLIST.md` — รายการตรวจคุณภาพเนื้อหา

> เป้าหมายของรอบนี้คือ "ยืนยันแหล่งข้อมูลทางการ" ของหน้ากฎหมายสำคัญ ไม่ใช่การตีความหรือสรุปกฎหมายใหม่
> **ห้ามเดา** URL หรือชื่อแหล่ง และห้ามอ้างว่าเป็นฉบับล่าสุดหากไม่ได้ตรวจจริง

## 1. ลำดับความน่าเชื่อถือของแหล่งทางการ

ตรวจจากแหล่งที่น่าเชื่อถือสูงสุดก่อนเสมอ:

1. **ราชกิจจานุเบกษา** (`ratchakitcha.soc.go.th`) — ตัวบทที่ประกาศอย่างเป็นทางการ ถือเป็นแหล่งอ้างอิงหลัก
2. **สำนักงานคณะกรรมการกฤษฎีกา** (`krisdika.go.th`) — ฉบับรวมที่แก้ไขเพิ่มเติม (consolidated) และคำอธิบาย
3. **หน่วยงานเจ้าของเรื่อง** เช่น กระทรวงศึกษาธิการ (`moe.go.th`), สำนักงาน ก.ค.ศ., คุรุสภา (`ksp.or.th`), สพฐ. (`obec.go.th`), สอศ., กสศ., กรมส่งเสริมการเรียนรู้ (สกร.) ฯลฯ
4. **เอกสารราชการ/ประกาศ** จากหน่วยงานที่เกี่ยวข้อง (ไฟล์ PDF ประกาศ ระเบียบ หรือหนังสือเวียนที่เผยแพร่ทางการ)

หลักการ: ถ้าหาได้ทั้งราชกิจจานุเบกษาและกฤษฎีกา ให้ใช้ **ราชกิจจานุเบกษาเป็น `source_url`** และอาจใส่ลิงก์กฤษฎีกาเพิ่มในรายการ `links`

## 2. วิธีตรวจสอบทีละหน้า

1. เปิด `docs/PRIORITY_LAW_SOURCE_CHECKLIST.md` แล้วเริ่มจากกลุ่ม **P1** ก่อน
2. นำชื่อกฎหมาย/ปี ไปค้นในแหล่งทางการตามลำดับข้อ 1
3. เปิดตัวบทจริงและตรวจว่า:
   - ชื่อกฎหมายตรงกับใน `data/laws.json`
   - ปี (`year`) และประเภท (`type`/`law_type`) ตรงกับฉบับจริง
   - หน่วยงาน (`agency`) สอดคล้องกับเจ้าของเรื่อง
   - มีฉบับแก้ไขเพิ่มเติมล่าสุดหรือไม่ (ถ้ามี ให้บันทึกไว้ แต่ **อย่าระบุว่าเป็นฉบับล่าสุด** หากไม่แน่ใจ)
4. คัดลอกลิงก์ที่ชี้ไปยัง **ตัวบทฉบับนั้นโดยตรง** (deep link) ไม่ใช่หน้าหลักของเว็บไซต์
5. บันทึกผลลง `SOURCE_REVIEW_LOG.md` (ดูข้อ 7)

---

## 3. กรอกอะไรลงใน `source_url`

- ใส่ลิงก์ที่ **เปิดอ่านได้จริงและชี้ไปยังฉบับนั้นโดยตรง** เช่น ไฟล์ PDF ราชกิจจานุเบกษาของฉบับนั้น หรือหน้ารวมฉบับในเว็บกฤษฎีกา
- ❌ **อย่าใส่** หน้าหลักทั่วไป เช่น `https://ratchakitcha.soc.go.th` เพียงอย่างเดียว — ในรอบตรวจนี้นับว่า *ยังไม่ผ่าน* (สถานะ `needs_source_url`)
- ถ้ายังหา deep link ไม่ได้ ให้ **ปล่อยว่างไว้** และคงสถานะ `needs_source_url` — อย่าใส่ลิงก์ปลอมหรือลิงก์ที่ไม่ตรงฉบับ

---

## 4. กรอกอะไรลงใน `official_source`

- ใส่ **ชื่อแหล่งทางการจริง** ที่เปิดอ่าน เช่น `ราชกิจจานุเบกษา`, `สำนักงานคณะกรรมการกฤษฎีกา`, `สำนักงานปลัดกระทรวงศึกษาธิการ`, `หอสมุดวุฒิสภา`
- ห้ามใช้ข้อความ placeholder `ต้องตรวจสอบจากแหล่งทางการ`
- ให้สอดคล้องกับ `source_url` (ถ้า `source_url` เป็นไฟล์ราชกิจจานุเบกษา ก็ควรระบุ `official_source` เป็น `ราชกิจจานุเบกษา`)

---

## 5. กรอกอะไรลงใน `last_checked`

- ใส่ **วันที่ที่เปิดตรวจแหล่งจริง** ในรูปแบบ `YYYY-MM-DD` (เช่น `2026-06-25`)
- หากไม่ระบุใน `data/laws.json` ระบบจะใช้ค่าวันที่เชิงโครงสร้างจาก `content/laws/_content.gotmpl` แทนบนหน้าเว็บ
  ซึ่ง **ไม่ใช่** การยืนยันว่ามีคนตรวจแหล่งจริง — รอบนี้จึงต้องเติม `last_checked` ของจริงเมื่อตรวจเสร็จ

---

## 6. การเขียน `status` ที่ปลอดภัย

ใช้ถ้อยคำที่ไม่กล่าวเกินจริง:

- `บังคับใช้` — เมื่อยืนยันว่ากฎหมายมีผลใช้บังคับแล้ว
- `อยู่ระหว่างพิจารณา` — สำหรับร่างกฎหมาย
- หลีกเลี่ยงคำกำกวมที่ระบบจะเตือน เช่น `ประกาศใช้`, `มีผลบังคับใช้` (ใช้ `บังคับใช้` แทน)
- หากไม่แน่ใจสถานะปัจจุบัน ให้คงข้อความเดิมและบันทึกข้อสงสัยใน `SOURCE_REVIEW_LOG.md` — **อย่าเปลี่ยนสถานะโดยไม่มีหลักฐาน**

ถ้อยคำมาตรฐานที่แนะนำสำหรับหมายเหตุ:

- "ตรวจสอบแหล่งข้อมูลทางการแล้ว"
- "ควรตรวจสอบฉบับแก้ไขเพิ่มเติมจากแหล่งทางการก่อนใช้อ้างอิง"
- "อยู่ระหว่างตรวจสอบแหล่งข้อมูล"

---

## 7. บันทึกผลลง `SOURCE_REVIEW_LOG.md`

เมื่อตรวจฉบับใดเสร็จ ให้เพิ่มแถวในตารางหัวข้อ **"รอบตรวจแหล่งข้อมูล ขั้นที่ 16"** โดยระบุ:

| คอลัมน์ | ใส่อะไร |
|---------|---------|
| วันที่ตรวจสอบ | วันที่เปิดตรวจจริง (`YYYY-MM-DD`) |
| ไฟล์/หน้า | `data/laws.json#<id>` |
| ชื่อกฎหมาย/เอกสาร | ชื่อเต็มตามฉบับจริง |
| แหล่งข้อมูลทางการที่ตรวจ | เช่น ราชกิจจานุเบกษา / กฤษฎีกา / หน่วยงานเจ้าของเรื่อง |
| source_url | ลิงก์เฉพาะฉบับที่เปิดอ่านจริง |
| ผลการตรวจสอบ | เช่น "ตรงกับฉบับ", "พบฉบับแก้ไขเพิ่มเติม", "ยังหา deep link ไม่ได้" |
| หมายเหตุ | ชื่อผู้ตรวจ/ข้อสังเกต |

---

## 8. สิ่งที่ "ห้ามทำ"

- ❌ เดา/แต่ง `source_url` หรือ `official_source`
- ❌ อ้างว่าเป็นฉบับล่าสุดโดยไม่ตรวจจริง
- ❌ แก้ความหมายของสรุปกฎหมาย (`summary` / `points`)
- ❌ เปลี่ยน `status` เป็นจำนวนมากโดยไม่มีหลักฐาน
- ❌ ลบ disclaimer หรือหมายเหตุการเรียนรู้
- ❌ แก้ไฟล์ที่สร้างอัตโนมัติ (`SOURCE_VERIFICATION_AUDIT.md`, `PRIORITY_LAW_SOURCE_CHECKLIST.md`) ด้วยมือ — ให้รัน `npm run audit:sources` ใหม่แทน

---

## 9. ขั้นตอนหลังแก้ `data/laws.json`

หลังเติม `source_url` / `official_source` / `last_checked` ของฉบับใด ๆ ให้รัน:

```bash
npm run audit:sources   # สร้างรายงาน/เช็กลิสต์ใหม่
npm run check:content    # ตรวจคุณภาพเนื้อหา
npm run report:stats     # ดูสรุปสถานะแหล่งข้อมูล
```

แล้วตรวจว่าสถานะของฉบับนั้นใน `SOURCE_VERIFICATION_AUDIT.md` เปลี่ยนเป็น `verified_official_source`
