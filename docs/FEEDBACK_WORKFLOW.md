# ขั้นตอนจัดการข้อเสนอแนะ (Feedback Workflow)

เอกสารนี้อธิบายวิธีรับ คัดกรอง และดำเนินการกับข้อเสนอแนะจากผู้ใช้สำหรับ **Education Law Hub** — ระบบแบบ static site ไม่มีฐานข้อมูลและไม่เก็บข้อมูลส่วนบุคคลโดยตรง

## ช่องทางรับข้อเสนอแนะ

| ช่องทาง | ผู้ใช้เป้าหมาย | หมายเหตุ |
|---------|----------------|----------|
| หน้า `/feedback/` | ผู้ใช้ทั่วไป | แนะนำ Google Form (ตั้งค่าใน `hugo.toml` → `params.feedbackGoogleForm`) |
| GitHub Issues | ผู้ใช้ทางเทคนิค / ผู้ร่วมพัฒนา | สาธารณะ — ใช้เทมเพลต `content-correction` หรือ `source-suggestion` |
| ลิงก์จากหน้ากฎหมาย | ผู้ที่พบปัญหาในหน้านั้น | ส่ง query `?page=` และ `url=` ไปยัง `/feedback/` |

ไม่มีการล็อกอิน ไม่มี backend — ข้อมูลทั้งหมดอยู่ในช่องทางภายนอก (Google / GitHub)

## ประเภทข้อเสนอแนะ

1. **แก้ไขเนื้อหา** — ข้อมูลผิด ล้าสมัย สรุปคลาดเคลื่อน ลิงก์เสีย
2. **เสนอแหล่งใหม่** — กฎหมาย/ประกาศที่ควรเพิ่มในคลัง
3. **ปรับปรุงเว็บไซต์** — UX ค้นหา แบบทดสอบ เส้นทางเรียนรู้
4. **ไม่ดำเนินการ** — คำปรึกษากฎหมายเฉพาะกรณี ข้อพิพาทส่วนบุคคน หรือไม่มีแหล่งทางการ

## การคัดกรอง (Triage)

ลำดับความสำคัญแนะนำ:

1. **ความปลอดภัยทางกฎหมาย** — สรุปที่อาจทำให้เข้าใจผิดในประเด็นสำคัญ
2. **ลิงก์แหล่งทางการเสีย** — แก้ได้เร็ว มีผลต่อความน่าเชื่อถือ
3. **ข้อมูลล้าสมัย** — เมื่อมีแหล่งยืนยันชัด
4. **เสนอแหล่งใหม่** — เมื่ออยู่ในขอบเขตการศึกษาและมีแหล่งทางการ
5. **ข้อเสนอ UX ทั่วไป** — จัดคิวตามความสะดวก

ขั้นตอนคัดกรอง:

- อ่านรายงาน — ตรวจว่ามี URL/ชื่อหน้าชัดหรือไม่
- ตรวจว่าเป็น **คำปรึกษากฎหมายเฉพาะกรณี** หรือไม่ → ตอบสุภาพว่าเว็บไซต์ไม่ให้บริการนี้ ไม่แก้เนื้อหาตามคำขอเฉพาะกรณี
- ตรวจว่ามี **แหล่งทางการ** แนบมาหรือไม่ — ถ้าไม่มี ขอเพิ่มก่อนดำเนินการ
- ตรวจว่ามี **ข้อมูลส่วนบุคคล** ใน Issue หรือไม่ → ขอให้แก้ไข/ลบ หรือปิด Issue หลังบันทึกเฉพาะส่วนที่จำเป็น

## การตรวจสอบแหล่งทางการ

1. เปิด `source_url` หรือลิงก์ที่ผู้แจ้งให้
2. ยืนยันว่าเป็นแหล่ง **รัฐหรือราชการ** (ราชกิจจานุเบกษา กฤษฎีกา เว็บหน่วยงาน)
3. เปรียบเทียบชื่อเรื่อง ปี พ.ศ. สถานะกับ `data/laws.json`
4. **ห้าม** แก้เนื้อหาตามความจำหรือแหล่งไม่เป็นทางการ
5. อ่าน [LEGAL_REVIEW_CHECKLIST.md](../LEGAL_REVIEW_CHECKLIST.md) และ [docs/CONTENT_EXPANSION_WORKFLOW.md](CONTENT_EXPANSION_WORKFLOW.md)

## การอัปเดตหน้ากฎหมายอย่างปลอดภัย

1. แก้ `data/laws.json` (วิธีหลัก) — ไม่เปลี่ยนความหมายทางกฎหมายโดยไม่มีแหล่ง
2. อัปเดต `last_checked` เป็นวันที่ตรวจจริง
3. แก้ `source_url` / `links` ถ้าลิงก์เสีย
4. ปรับสรุปเฉพาะเมื่อแหล่งทางการรองรับ — ใช้ภาษาระมัดระวัง (“โดยทั่วไป” “ตามที่ปรากฏในแหล่งทางการ”)
5. รัน `npm run check:content` และ `npm run build:search`
6. ตรวจหน้า `/laws/<id>/` บนเครื่อง

## บันทึกใน SOURCE_REVIEW_LOG.md

ทุกครั้งที่แก้เนื้อหาหรือแหล่งจาก feedback:

```markdown
| วันที่ | รายการ | การดำเนินการ | ผู้ตรวจ | หมายเหตุ |
|--------|--------|--------------|---------|----------|
| 2026-06-24 | พ.ร.บ. ... | แก้ source_url ตาม Issue #12 | ชื่อ | ลิงก์ รก. เดิมเสีย |
```

## การปิด Issue

- **แก้แล้ว** — อ้างอิง commit/PR และลิงก์หน้าที่อัปเดต
- **ไม่ดำเนินการ** — อธิบายสั้นๆ (นอกขอบเขต ไม่มีแหล่งทางการ ซ้ำกับที่มีแล้ว)
- **ต้องการข้อมูลเพิ่ม** — ใช้ label หรือคอมเมนต์ขอแหล่งทางการ แล้วปิดถ้าไม่มีการตอบกลับในระยะที่กำหนด

## สิ่งที่ไม่ควรจัดการเป็นคำปรึกษากฎหมาย

- คำถาม “ฉันถูกหรือผิดในกรณี X”
- คำขอให้ตีความกฎหมายเพื่อใช้ในคดีหรือข้อพิพาท
- รายละเอียดส่วนตัวของเด็ก ครู หรือผู้ปกครองในสถานการณ์เฉพาะ

ตอบอย่างสุภาพว่าเว็บไซต์จัดทำเพื่อการศึกษาเบื้องต้น แนะนำปรึกษาหน่วยงานที่เกี่ยวข้องหรือผู้เชี่ยวชาญตามความเหมาะสม

## ข้อควรระวังด้านความเป็นส่วนตัว

- อย่าเก็บสำเนา Google Form ที่มีข้อมูลส่วนบุคคลใน repo
- อย่าคัดลอกข้อมูลส่วนบุคคลจาก Issue ลงใน commit
- ถ้าพบข้อมูลอ่อนไหวใน Issue สาธารณะ — ขอให้ผู้แจ้งแก้ไขหรือลบ และดำเนินการตามนโยบาย GitHub

## การตั้งค่า Google Form

เมื่อมีแบบฟอร์มจริง:

1. สร้าง Google Form ฟิลด์ขั้นต่ำ: ประเภท, ชื่อหน้า/URL, รายละเอียด, ลิงก์แหล่งทางการ (ไม่บังคับชื่อ-นามสกุล)
2. ใส่ URL ใน `hugo.toml`:

```toml
[params]
  feedbackGoogleForm = 'https://docs.google.com/forms/d/e/.../viewform'
```

3. ทดสอบหน้า `/feedback/` — ปุ่มแบบฟอร์มจะแสดงแทนข้อความ TODO
